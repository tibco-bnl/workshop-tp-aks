# Root Cause Analysis: Login Failures in v1.15.0

## Issue Summary

After deploying TIBCO Platform Control Plane v1.15.0, admin login failed with "Internal Server Error" despite:
- ✅ CoreDNS configured correctly
- ✅ OAuth callbacks succeeding
- ✅ User authentication working
- ✅ Database records manually corrected (user activated, team created, team membership assigned)

## Root Cause

The `values/cp1-values.yaml` file contained **two undocumented parameters** that are **NOT supported in v1.15.0**:

```yaml
# ❌ These parameters caused the issue:
adminInitialPassword: "Welcome123!"
enable_api_based_initialization: true
```

### Why This Caused Problems

**In v1.15.0, the admin password and subscription initialization process changed:**

| Aspect | v1.14 | v1.15 |
|--------|-------|--------|
| Admin Password | User-defined via `adminInitialPassword` | **Auto-generated** |
| Password Storage | Database only | Secret `tp-cp-web-server` |
| Initialization Flag | Optional `enable_api_based_initialization` | **Always enabled** (parameter removed) |
| Subscription Creation | Manual or flag-based | **Automatic during deployment** |

**What Happened:**
1. v1.15 code likely ignored the unsupported `adminInitialPassword` parameter
2. v1.15 tried to auto-generate a password, but the parameter conflict caused incomplete initialization
3. Database schema was partially created, but critical linking tables were missing
4. Manual SQL fixes (activating user, creating team, setting membership) couldn't fix the underlying initialization failure

### Evidence

**From v1.15 Official Setup Guide (Section 10.1):**
```bash
# Retrieve auto-generated admin password
export TP_ADMIN_PASSWORD=$(kubectl get secret tp-cp-web-server \
  -n ${TP_CP_NS} \
  -o jsonpath='{.data.TSC_ADMIN_PASSWORD}' | base64 --decode)
```

**Neither parameter documented in v1.15 guide:**
- No mention of `adminInitialPassword` anywhere in v1.15 documentation
- No mention of `enable_api_based_initialization` in v1.15 documentation

**From user's failed deployment:**
```bash
$ kubectl get secret tp-cp-web-server -n cp1-ns -o jsonpath='{.data.TSC_ADMIN_PASSWORD}'
Error from server (NotFound): secrets "tp-cp-web-server" not found
```
↳ Secret wasn't created because initialization failed

## The Fix

### 1. Correct the Values File

**File**: `values/cp1-values.yaml`

**BEFORE (Wrong):**
```yaml
global:
  external:
    # ❌ WRONG - Unsupported in v1.15
    enable_api_based_initialization: true
    
    # ❌ WRONG - Unsupported in v1.15
    adminInitialPassword: "Welcome123!"
    
    admin:
      customerID: "dp1-customer"
      email: "admin@tibco.com"
      firstname: "Platform"
      lastname: "Admin"
```

**AFTER (Correct):**
```yaml
global:
  external:
    # ✅ CORRECT - No password, no initialization flag
    admin:
      customerID: "dp1-customer"
      email: "admin@tibco.com"
      firstname: "Platform"
      lastname: "Admin"
```

### 2. Clean Database and Redeploy

**Step 1: Drop corrupted databases**
```bash
./04a-cleanup-databases.sh
```

This drops all `cp1_*` databases to remove the incomplete schema created by the failed initialization.

**Step 2: Redeploy with corrected values**
```bash
./04-install-controlplane.sh
```

The corrected values file will allow v1.15 to properly:
- Auto-generate admin password
- Create complete database schema
- Initialize subscription automatically
- Set up proper user/team/account linking

**Step 3: Retrieve auto-generated password**

## Lessons Learned

### 1. Always Consult Version-Specific Documentation
- Don't carry over values files from v1.14 to v1.15 without reviewing changes
- Check official setup guides for the EXACT version you're deploying

### 2. Understand Breaking Changes
- v1.15.0 introduced significant changes to admin user initialization
- Parameters that worked in v1.14 may be deprecated or cause conflicts in v1.15

### 3. Trust the Auto-Generation
- v1.15's auto-generated passwords are more secure
- Manual password setting is no longer supported (and for good reason)

### 4. Database Fixes Won't Work for Initialization Issues
- Manually correcting database records (like we did with isActive, team membership) doesn't fix initialization failures
- If the deployment didn't complete correctly, you need to redeploy with correct configuration
